# OpenFusion Roadmap — Milestone Sequence

**Source spec:** `docs/superpowers/specs/2026-07-03-harness-fusion-app-design.md`

Each milestone is a separate implementation plan in this directory, written
**just-in-time** when the previous milestone completes (so plans reflect
current SDK/protocol realities, not stale guesses). Every milestone ends
with working, independently testable software. This roadmap is the
persistent work queue for `/loop`-driven development: each loop iteration
executes the next unchecked task in the current milestone plan; when a plan
finishes, the next iteration writes the next milestone's plan from the spec
+ this roadmap.

| # | Milestone | Deliverable (exit criteria) | Spec sections |
|---|---|---|---|
| M0 | **Scaffold + engine core** | pnpm/cargo-ready monorepo; `@openfusion/engine` boots as a process, answers `engine.ping`/`engine.info` over ndjson JSON-RPC on stdio; CI green on GitHub Actions. Plan: `2026-07-03-m0-scaffold-engine-core.md` | §4.1, §9, §10 |
| M1 | **Wiki — symbol layer** | Point engine at a repo → tree-sitter symbol graph + token-budgeted repo map (PageRank); SQLite store; git-SHA-watermark incremental refresh; `engine.wiki.build`/`engine.wiki.query` RPC; served via in-engine MCP server. | §4.2 wiki/, store/ |
| M2 | **Models layer** | Vercel AI SDK providers: Moonshot, Z.ai, DeepSeek presets + generic OpenAI-compatible; pricing table; per-call cost meter; fallback chains. `engine.models.complete` fixture-tested, no live keys in CI. | §4.2 models/, §5.6 |
| M3 | **Frontier engines** | `FrontierEngine` interface (ACP-shaped); Claude Code adapter via Agent SDK with wiki MCP attached; session lifecycle (start/prompt/stream/stop). Exit: frontier session answers repo questions using the wiki. Codex CLI adapter is M3.5 (fast-follow, same interface). | §4.2 engines/ |
| M4 | **Harnessgen** | Understanding phase: frontier-driven repo exploration → `.openfusion/` artifacts (manifest, wiki prose pages + digests, agents, routing.yaml) validated against schema; exporters (AGENTS.md, Claude Code subagents). Exit: `engine.harness.generate` on a sample repo. | §4.2 harnessgen/, §4.3 |
| M5 | **Orchestrator runtime** | Task decomposition + classification; routing per routing.yaml; minimal worker loop (bash + edit tools) on open models; worktree isolation; frontier review gate; retry-once-then-escalate. Exit: one end-to-end task on a sample repo, open-model worker + frontier review. | §5, §8 |
| M6 | **Evals** | Repo-derived micro-evals (build, test subsets, golden tasks from recent commits); baseline-vs-harness comparison; report card. Exit: `engine.evals.run` produces a report card for a generated harness. | §6 |
| M7 | **Desktop shell** | Tauri 2 app: onboarding, understanding progress, cockpit (chat/task tree/worker cards/diff review/cost meter), harness editor, settings; Keychain secrets; sidecar supervision + crash recovery. | §4.1 shell, §7, §8 |
| M8 | **Distribution** | Developer ID signing + notarization, DMG on GitHub Releases via tauri-action, auto-update, Homebrew cask; public naming decision; v0.1.0 release. | §10 |

**Ordering rationale:** engine-first (M0–M6) keeps everything testable
headless and preserves the future standalone-CLI option; the shell (M7)
lands only once there is a real engine to supervise; distribution (M8) is
last because signing/notarization needs the Apple Developer account and a
stable artifact.

**Standing risks to re-check at each plan-writing point:** Agent SDK /
Codex app-server API churn (§12.3); ETH generated-context hazard gates M4
behind M6-style smoke evals even before M6 proper (§12.1).
