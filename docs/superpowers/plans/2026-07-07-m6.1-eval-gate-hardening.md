# M6.1: Eval-Gate Hardening — Two-Dimensional & Significance-Aware Verdict

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Sharpen the *existing, fully-implemented* `engine.evals.run` verdict (packages/engine/src/evals/run.ts) so its pass/fail conclusion is honest under three findings from the 2026-07-07 harness-composition research: (1) single-run pass@1 is noisy (~5pp band), so a within-noise quality wobble must not flip the manifest to "fail"; (2) a savings *claim* needs materially more tasks than a hazard *flag*; (3) the gate is two-dimensional — a harness that holds quality but costs materially *more* is an ETH **cost** hazard, not a silent "inconclusive".

**Architecture:** Every change lives in ONE function — `runEvals`'s verdict computation (run.ts ~lines 800–953) — plus three new module-level constants and a handful of new report-card fields, all additive. **Tech Stack:** All in-codebase (evals/run.ts, evals/tasks.ts fixtures, harness/store.ts's `harnessStatus`/`setEvalsVerdict`). Vitest. No new dependencies.

**Source of the deltas:** `docs/research/2026-07-07-harness-composition.md` §3.3 (randomness/noise, arXiv:2602.07150), §4.2 (task-floor math), §4.3 (two-dimensional verdict), §0 (ETH cost hazard, arXiv:2602.11988).

### Task 2: significance guard — a within-noise quality gap is not an ETH hazard

**Why:** Research §3.3 (arXiv:2602.07150) — single-run pass@1 has std >1.5pp even at temperature 0; deltas under ~5pp are inside the noise band. The current ETH-hazard branch fires whenever `cleanHarnessPassed < cleanBaselinePassed` by even ONE task — so on a large suite a 1-task wobble (e.g. 1/40 = 2.5pp) wrongly flips the manifest to `"fail"`. Require the clean-subset quality gap to *exceed the noise band* before calling it a genuine hazard. On small suites any gap is a large fraction, so this preserves aggressive flagging exactly where it's warranted (the existing 2-task ETH test still fails, correctly).

**Files:**
- Modify: `packages/engine/src/evals/run.ts` (new constant; guard the ETH-hazard branch; new report field + note)
- Test: `packages/engine/test/evals-run.test.ts` (new partial-failure fixture + two bracketing tests)

**Interfaces:**
- Consumes: `cleanBaselinePassed`, `cleanHarnessPassed`, `cleanTasks.length` (all already computed in `runEvals`).
- Produces: a new constant `QUALITY_NOISE_BAND = 0.05`; a new boolean `qualityGapIsSignificant` used to guard the `!qualityHeldClean` branch; a new additive report-card field `qualityGapWithinNoise: boolean`. A new test fixture `makePartialFailFrontierAdapter` (fails the harness side on its first N invocations; baseline always correct) so a small clean-subset gap is producible.

- [ ] **Step 1: Add the constant.** In run.ts, after `MATERIAL_MEASUREMENT_FAILURE_FRACTION` (~line 260), add:

```ts
// Research 2026-07-07 (§3.3, arXiv:2602.07150): single-run pass@1 varies
// 2.2–6.0pp and has std >1.5pp even at temperature 0, so a clean-subset
// quality gap SMALLER than this fraction of the clean task count is
// indistinguishable from measurement noise and must NOT be reported as an
// ETH-hazard "fail". Above it, the gap is treated as a genuine quality
// regression. On tiny suites any gap is a large fraction (e.g. 1/2 = 50pp),
// so aggressive flagging is preserved exactly where the sample is too small
// to explain the gap as noise. Single-run heuristic; multi-run averaging to
// buy significance for smaller effects is a deferred follow-up.
const QUALITY_NOISE_BAND = 0.05;
```

- [ ] **Step 2: Add the partial-failure fixture (test file).** In evals-run.test.ts, after `makeFakeEvalsFrontierAdapter` (~line 180), add:

```ts
// Like makeFakeEvalsFrontierAdapter, but the HARNESS side (resultLabel !=
// "eval-baseline") writes the WRONG source on its first `harnessFailCount`
// invocations and the correct source thereafter; the baseline side is always
// correct. The frontier-only fixture harness routes straight to escalation —
// exactly ONE frontier session per task on each side — so the harness-side
// counter maps 1:1 to task order, letting a test produce a SMALL clean-subset
// quality gap (e.g. 1 of 30) to exercise QUALITY_NOISE_BAND.
function makePartialFailFrontierAdapter(opts: {
  harnessFailCount: number;
  baselineCostUsd: number;
  harnessCostUsd: number;
  meter: CostMeter;
}): FrontierAdapter {
  let harnessSeen = 0;
  return {
    kind: "claude-code",
    async createSession({ projectDir, resultLabel }): Promise<FrontierSession> {
      const isBaseline = resultLabel === "eval-baseline";
      let correct: boolean;
      if (isBaseline) {
        correct = true;
      } else {
        harnessSeen += 1;
        correct = harnessSeen > opts.harnessFailCount; // first N harness runs fail
      }
      const costUsd = isBaseline ? opts.baselineCostUsd : opts.harnessCostUsd;
      return {
        id: randomUUID(),
        projectDir,
        prompt(_text: string, _promptOpts?: { timeoutMs?: number }): FrontierPromptHandle {
          async function* gen(): AsyncGenerator<FrontierEvent> {
            writeFileSync(path.join(projectDir, "source.js"), correct ? CORRECT_SOURCE : WRONG_SOURCE);
            yield { type: "tool_use", name: "Write", summary: "wrote source.js" };
            yield { type: "text", text: "done" };
            const event: FrontierEvent = {
              type: "result",
              resultText: "done",
              costUsd,
              usage: { inputTokens: 10, outputTokens: 5, cacheReadTokens: 0 },
              numTurns: 1,
              durationMs: 1,
              engineSessionId: null,
            };
            opts.meter.record({
              providerId: "claude-code",
              kind: "frontier-claude",
              model: "fake-frontier-model",
              usage: event.usage,
              costUsd: event.costUsd,
              at: Date.now(),
              source: isBaseline ? "frontier-review" : "frontier-escalate",
              pricingConfidence: costUsd !== null ? "verified" : "unpriced",
            });
            yield event;
          }
          return { events: gen(), abort: () => {} };
        },
        async close(): Promise<void> {},
      };
    },
  };
}
```

### Task 3: two-dimensional verdict — a material cost regression at held quality is an ETH cost hazard

**Why:** Research §0/§4.3 (arXiv:2602.11988) — the ETH failure mode is *quality held but cost UP ~20%*. The current verdict treats "quality held, savings ≤ 0" as a silent `"inconclusive"`; a *material* cost regression should be flagged as a `"fail"` (the harness is actively failing its cost-savings purpose while paying more). Mild increases stay inconclusive.

**Files:**
- Modify: `packages/engine/src/evals/run.ts` (new constant; new verdict branch)
- Test: `packages/engine/test/evals-run.test.ts` (two tests bracketing the threshold)

**Interfaces:**
- Consumes: `qualityHeldClean`, `qualityGapWithinNoise` (Task 2), `cleanSavingsPct`, `taskCount`, `MIN_TASK_COUNT_FOR_VERDICT` (5).
- Produces: a new constant `COST_REGRESSION_FAIL_FRACTION = 0.10`; a new verdict branch placed AFTER the unpriced gate and BEFORE the savings-pass floor gate, flipping the manifest to `"fail"` on a material cost regression at held quality.

- [ ] **Step 1: Add the constant.** In run.ts, after `QUALITY_NOISE_BAND` (Task 2), add:

```ts
// Research 2026-07-07 (§0/§4.3, arXiv:2602.11988): the eval gate is
// two-dimensional. A harness that HOLDS quality but costs materially MORE
// than the no-harness baseline is an ETH COST hazard (the study's exact
// failure mode: quality held, cost +~20%), not a neutral result. When the

// (MIN_TASK_COUNT_FOR_VERDICT) not the savings-PASS floor — a harm signal is
// flagged readily, asymmetric with the higher bar a positive claim must clear.
const COST_REGRESSION_FAIL_FRACTION = 0.10;
```
