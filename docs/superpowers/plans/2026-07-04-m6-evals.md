# M6: Evals — Measure Cost at Held Quality — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `engine.evals.run {projectDir}` runs repo-derived micro-evals baseline-vs-harness and produces a report card: for each eval task, run it once with the FRONTIER doing it directly (baseline) and once through the full orchestrate loop (harness), score both by whether the repo's tests pass, and measure cost by surface. The report card reports pass rates and the cost delta, flips `manifest.verification.evals` pending→pass/fail, and — per the ETH hazard (spec §12.1) — FLAGS a harness whose quality drops below baseline rather than passing it. Plus the pre-eval prerequisites the M5b review made hard gates: refreshed/confidence-tagged pricing, real-adapter timeout so an eval batch can't wedge, and wiki digests actually reaching worker prompts.

**Architecture:** Task 0 refreshes pricing (the savings numbers depend on it). Task 1 makes the real Claude adapter honor `timeoutMs` + registers adapter-direct sessions in FrontierService bookkeeping (so `engine.close`/deadlines can reach review/escalate/generate sessions — an eval batch of N real loops can't hang on one wedge). Task 2 wires wiki digests into worker prompts (the harness's headline ingredient — otherwise M6 measures the harness without its main feature) + exposes `taskClass` and the review/escalate cost split on `OrchestrateResult`. Task 3 is `src/evals/tasks.ts` (eval-task model + the repo-tests oracle runner + golden-task-from-commit construction). Task 4 is `src/evals/run.ts` + `methods.ts` (`engine.evals.run` — baseline vs harness, score, aggregate, report card, flip manifest). Task 5 is docs + report-card format.

**Tech Stack (verified 2026-07-04, docs/research/2026-07-04-m6-pricing-eval-verification.md):** all in-codebase (orchestrate, worker, frontier, meter, worktree). Oracle = run the repo's own test command in a worktree, check exit code (hand-rolled, pragmatic). Golden tasks = SWE-bench-style commit reversion. No new deps.

## Global Constraints

- Everything standing: Node ≥22, strict TS NodeNext `.js` imports, tsconfig.test coverage, stdout protocol purity, tmp git-repo fixtures, conventional commits + `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`, nothing under `.superpowers/`/`.claude/` of THIS repo, auth-agnostic, prompts/model-output/file-content never logged.
- **No live keys / no claude binary in CI:** the eval RUNNER mechanics (construct task → baseline → harness → score → aggregate) are tested with FAKE models (fake worker + fake frontier) over SYNTHETIC fixture eval tasks (tiny git repos with a known change + a test that fails-before/passes-after). The REAL repo-mined 20–50-task eval run is an env-gated operator smoke.
- **Savings-number integrity:** the report card carries the pricing confidence of the models it priced; any unverified-priced model taints its cost figure (report card marks it). No savings number ships without pricing confidence attached.
- **ETH hazard (spec §12.1):** a harness whose pass rate is materially below baseline is FLAGGED (`verification.evals = "fail"`), never silently "passed".
- **Git-history leak mitigation:** golden-task worktrees must not expose the answer via `git log --all`/reflog/other branches (the target commit and future history must be unreachable from the eval checkout).

### Task 3: eval task model + repo-tests oracle + golden-task construction

**Files:**
- Create: `packages/engine/src/evals/tasks.ts`
- Test: `packages/engine/test/evals-tasks.test.ts`

**Interfaces:**
- `interface EvalTask { id: string; prompt: string; setup: (worktreeRoot: string) => Promise<void>; testCommand: string[] }` — `setup` mutates a fresh worktree into the pre-change state; `prompt` is the task text; `testCommand` is the repo's test command (e.g. `["pnpm","test"]` or `["node","--test"]`) run in the worktree to score.
- `runOracle(worktreeRoot: string, testCommand: string[], timeoutMs?: number): Promise<{ passed: boolean; exitCode: number; durationMs: number }>` — execFile the test command in the worktree (cwd), timeout, return pass = exitCode 0. Never shell. Truncate captured output; do NOT log it.
- `goldenTaskFromCommit(repoDir: string, commitSha: string): Promise<EvalTask>` — SWE-bench-style: the task's `setup` produces a worktree at the commit's PARENT (`<sha>^`) with git history stripped so the answer isn't reachable (checkout the parent tree into a fresh NON-git dir, OR a shallow worktree with future history/branches removed — document the mechanism; the key invariant: the eval checkout cannot `git log --all`/reflog to read the target commit). `prompt` derived from the commit message/subject ("Implement: <subject>"). `testCommand` = the repo's configured test command. NOTE the gotchas from research: prefer few-file commits with their own tests; caller supplies the commit list (a picker heuristic can be a later improvement). goldenTaskFromCommit constructs ONE task from a given sha; task SELECTION (which commits) is the caller's / operator-smoke's job in v1.
- `synthEvalTask(...)` test helper: a tiny fixture — a worktree with a file and a test that FAILS before the change and PASSES after; the `prompt` describes the change. Used by CI tests (deterministic, no real model).

- [ ] **Step 1: Failing tests** (real tmp git repos): runOracle returns passed=true on a repo whose test command exits 0, false on nonzero, respects timeout; goldenTaskFromCommit on a fixture repo with a 2-commit history (commit adds a feature + its test) → setup produces the parent state (feature absent, test present-and-failing OR test added by the commit — handle both: if the commit ALSO adds the test, the parent lacks the test so the oracle can't run → document that v1 golden tasks require the test to pre-exist at the parent, i.e. fail-to-pass on an existing test; a commit that adds both code and its test is skippable or the test is applied separately — pick and document), and the checkout can't reach the target commit via git log; synthEvalTask fails-before/passes-after asserted.
- [ ] **Step 2: RED → implement → GREEN** (exact totals) → Commit `feat(engine): eval task model, repo-tests oracle, golden-task-from-commit construction`

### Task 4: engine.evals.run — baseline vs harness report card

**Files:**
- Create: `packages/engine/src/evals/run.ts`, `packages/engine/src/evals/methods.ts`
- Modify: `packages/engine/src/engine.ts` (EvalsService + re-exports), `packages/engine/src/harness/store.ts` (flip manifest.verification.evals)
- Test: `packages/engine/test/evals-run.test.ts`; env-gated `evals-run-smoke.test.ts`

**Interfaces:**
- `EvalsService` on Engine composing OrchestrateService + FrontierService + WorktreeManager + ModelsService + HarnessService.
- `engine.evals.run { projectDir, tasks?: <task descriptors>, sampleNote?: string }` → a report card:
  ```
  { taskCount: number,
    baseline: { passed: number; costUsd: number|null },
    harness:  { passed: number; costUsd: number|null; escalations: number },
    savingsPct: number|null,           // (baseline.costUsd - harness.costUsd)/baseline.costUsd, null if either unpriced
    qualityHeld: boolean,              // harness.passed >= baseline.passed (within tolerance)
    verdict: "pass" | "fail" | "inconclusive",  // pass = qualityHeld && savings>0 && enough tasks; fail = quality dropped; inconclusive = too few tasks / unpriced
    pricingConfidence: string,         // worst across all cost records — taints the savings claim
    perTask: Array<{ id; baselinePassed; harnessPassed; harnessOutcome; baselineUsd; harnessUsd }>,
    note: string }                     // caveats: sample size, estimate-class, pricing confidence
  ```
- Flow per task: create TWO fresh worktrees (baseline + harness) via the task's `setup`. BASELINE: run the task with the FRONTIER directly in the baseline worktree (a frontier session with writeScope = worktree, like escalation-as-primary — reuse orchestrate's escalation primitive or a thin frontier-does-task helper), apply its diff... actually simpler: baseline and harness each produce a diff; APPLY it into the worktree, run the oracle (testCommand) → passed. Score baseline.passed / harness.passed. HARNESS: `engine.orchestrate` on the harness worktree → diff → apply → oracle. Sum costs by surface (baseline = pure frontier cost; harness = worker + review + escalate). Aggregate. Cleanup worktrees (both are transient eval scratch — remove after scoring; this is eval, not the user's work, so auto-remove is correct here UNLIKE orchestrate).
- Report-card verdict logic: `qualityHeld` = harness.passed >= baseline.passed (M6 v1 tolerance: harness not worse). `savingsPct` computed only if both costs priced (else null → verdict inconclusive). Sample-size: < a threshold (e.g. 5) → "inconclusive" regardless (a demo, not a claim — cite the note). ETH hazard: harness.passed < baseline.passed → verdict "fail" (harness DEGRADES quality — the exact thing evals exist to catch).
- `store.ts`: a `setEvalsVerdict(projectDir, verdict: "pass"|"fail"|"pending")` that updates ONLY `manifest.verification.evals` (re-write the manifest with the new value, preserving everything else + the artifacts list + manifest-last atomicity). engine.evals.run flips it after a run (pass/fail; inconclusive leaves pending).
- CI test: fake worker (setTestModel — a worker that "solves" synthetic tasks by writing the fix) + fake frontier (reviews + escalates); run over 2-3 synthEvalTasks → report card computed, verdict logic exercised (a run where harness matches baseline → pass; a run where the fake harness FAILS a task baseline passed → verdict "fail" [ETH-hazard test]; a too-few-tasks run → inconclusive; an unpriced model → savings null + inconclusive/note). manifest.verification.evals flipped. Env-gated real smoke: real golden tasks mined from THIS repo's recent commits, real open-model worker + real frontier — `it.skipIf(!process.env.OPENFUSION_EVALS_SMOKE)`.

- [ ] **Step 1: Failing tests → Step 2: RED → implement → GREEN** (exact totals) → Commit `feat(engine): engine.evals.run — baseline-vs-harness report card with ETH-hazard gate`
